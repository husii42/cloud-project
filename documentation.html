Cloud and DevOps Engineering: Part I & II

Aalen University  ·  Infrastructure as Code with Terraform on Azure  ·  Hüseyin Simsek  ·  Summer Semester 2026

1. What is this?

This project provisions a cloud infrastructure on Microsoft Azure using Terraform, an Infrastructure as Code (IaC) tool, and deploys a small Flask web application on top of it. Part I defines the infrastructure; Part II adds the application code, the identity-based access between the app and its resources, and a CI/CD pipeline that builds and deploys the app automatically.

The following Azure resources are created:

ResourceNamePurpose
Resource Grouprg-<name>-devContainer for all resources
Storage Accountst<name>devStores uploaded images
Storage ContainerimagesBlob container inside the Storage Account
Key Vaultkv-<name>-devStores sensitive credentials securely
App Service Planasp-<name>-devThe underlying compute for the web app
Web Appapp-<name>-devHosts the Flask web application

The web application itself provides two pages:

PageRoutePurpose
Web Page 1/Lists all files in the Storage Account, each with a direct download link
Web Page 2/uploadUpload form to add a new file/image to the Storage Account

How to Run (in PowerShell)

# 1. Clone the repository
git clone https://github.com/husii42/cloud-project.git
cd cloud-project

# 2. (Windows only) Add Terraform to PATH if not recognised
$env:path += ";C:\Program Files\Terraform"

# 3. Create the variables file and set a unique project_name
cp terraform.tfvars.example terraform.tfvars
notepad terraform.tfvars

# 4. Authenticate with Azure
az login

# 5. Initialise Terraform (downloads the Azure provider)
terraform init

# 6. Preview what will be created
terraform plan

# 7. Deploy the infrastructure
terraform apply
The project_name in terraform.tfvars must be globally unique and contain only lowercase letters and numbers — it becomes part of the Storage Account name (e.g. stmyname42dev).

Deploying the application

The application itself is not deployed by Terraform directly. Instead, a GitHub Actions pipeline builds and deploys it automatically whenever the app/ directory changes on the master branch. This requires a one-time setup of an OIDC trust relationship between GitHub and Azure, described in full in deployment-notes.md. The short version: 

# 1. Register an Azure AD application for GitHub Actions to authenticate as
az ad app create --display-name "github-cloud-project-deploy"
az ad sp create --id <appId>

# 2. Grant it permission to deploy to the resource group
az role assignment create --assignee <appId> --role "Website Contributor" `
  --scope /subscriptions/<sub-id>/resourceGroups/rg-<name>-dev

# 3. Create federated credentials so GitHub tokens are trusted (no stored secret)
az ad app federated-credential create --id <appId> --parameters '{...subject: repo:.../ref:refs/heads/master...}'
az ad app federated-credential create --id <appId> --parameters '{...subject: repo:.../environment:production...}'

# 4. Add AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_SUBSCRIPTION_ID as GitHub repository secrets
Two federated credentials are needed, not one: the pipeline's deploy job is tied to a GitHub environment (production), which changes the subject string GitHub presents to Azure AD compared to a plain branch push. Without the second credential, the login step fails with AADSTS700213: No matching federated identity record found.

How the files work together

terraform.tfvars provides the values (project name, region, environment). variables.tf defines which values are expected. main.tf passes those values into each module and connects their outputs — for example, the Storage Account access key is read from the storage module and passed directly into the Key Vault module, which stores it as a secret. The Web App's Managed Identity principal_id is passed from the appservice module into both storage and keyvault, so those modules can grant it access. After deployment, outputs.tf displays all relevant resource names and URLs in the terminal. The app/ directory holds the Flask application that runs on top of this infrastructure, and .github/workflows/deploy.yml builds and deploys it.

4. Approach & Reasoning

Modular structure

The infrastructure is split into three modules (Storage, Key Vault, App Service) rather than placing everything in a single main.tf file. Each module is responsible for exactly one concern: it receives input variables, creates its resources, and exposes outputs. The root main.tf acts as the orchestrator: it passes values into each module and connects their outputs where needed.

This separation had a practical benefit in Part II: nothing in the existing Part I resources needed to be replaced. The new role assignment, access policy, and app settings were simply added on top of the existing modules. Had everything been written in one file, adding Part II would have meant editing existing resources directly, which risks unintended changes or data loss.

Resolving the module dependency between App Service, Storage and Key Vault

Granting the Web App's Managed Identity access to Storage and Key Vault creates a two-directional need: the storage and keyvault modules need the Web App's principal_id to create their role assignment / access policy, while the appservice module needs the Key Vault URI and Storage Account name to configure its app settings. Referencing those values directly as module outputs in both directions would create a circular dependency between modules (appservice → keyvault → appservice), which Terraform cannot resolve.

The dependency is broken by computing the Key Vault URI and Storage Account name in the root module directly from the shared naming convention (project_name / environment), rather than reading them back from the other modules' outputs. The appservice module therefore only depends on the Resource Group, so its principal_id is available immediately for storage and keyvault to consume.

Key Vault for sensitive data

The Storage Account access key is a sensitive credential: anyone who holds it has full read and write access to the storage. Storing it directly in application code or committing it to Git would expose it to anyone with access to the repository.

Terraform reads the access key from the Storage Account after creation and stores it automatically as a secret in Azure Key Vault. Access to the Key Vault is controlled via Access Policies: the developer has full access during deployment, and the Web App's Managed Identity is granted read-only (Get, List) access. In practice, the application does not currently read this secret at runtime — it authenticates to Storage directly via its Managed Identity — but the access policy demonstrates that the same identity model extends uniformly to Key Vault as well.

Managed Identity instead of passwords

A common approach for giving an application access to Azure resources is to create a service principal with a client secret (essentially a username and password). The problem with this approach is that the secret needs to be stored somewhere: in environment variables, configuration files, or a pipeline, which creates another credential that can be accidentally exposed.

By assigning the Web App a System-Assigned Managed Identity and granting it the Storage Blob Data Contributor role (scoped to read/write/delete blobs only, not account keys or account-level settings), Azure manages the authentication automatically. The application authenticates via the azure-identity SDK's DefaultAzureCredential, which uses the Managed Identity when running on Azure and falls back to the developer's own az login session locally — so no password is ever created, stored, or rotated manually, and the same code works in both environments.

Full infrastructure defined in Part I

The App Service, Key Vault, and Storage Account are all provisioned in Part I, even though the application code was not written until Part II. These resources form the foundation the web application needs to run securely in the cloud.

Provisioning everything upfront also avoids a practical problem with Terraform: if a resource is created, used to store data, and then needs to be replaced in a later apply, Terraform must destroy the existing resource first. This would delete any data already stored in it. By defining the full infrastructure from the beginning, Part II could extend what was already there — adding a role assignment, an access policy, and app settings — without triggering any destructive changes (confirmed via terraform plan: 2 to add, 1 to change, 0 to destroy).

Upload authentication

The Storage Account permits GET, POST and PUT requests from any origin via CORS, and the application's /upload route has no login or access control of its own. This is intentional at both the infrastructure and application level: the Storage Account itself does not need to know who is allowed to upload files, because uploads do not go directly from the browser to Azure Storage. Instead, the request flows through the App Service, which acts as an intermediary: 

Browser → App Service (Flask, Managed Identity) → Storage Account

The App Service is the only component that communicates directly with the Storage Account, using its own identity — the browser never receives or needs any Azure credential. Deciding whether a specific user is allowed to upload is an application-level concern that was deliberately left out of scope for this project (see Known Limitations): anyone with the application URL can currently upload and download files. Terraform governs access at the Azure infrastructure level; the application would govern access at the user level if user-facing authentication were added.

CI/CD without stored secrets

The GitHub Actions pipeline (.github/workflows/deploy.yml) builds, tests, and deploys the application in three jobs: build (install dependencies, compile-check, run smoke tests against the Flask routes, package the app), deploy (log in to Azure and push the package to the Web App), and verify (poll the deployed /healthz endpoint to confirm the deployment actually succeeded).

Authentication from GitHub Actions to Azure uses OIDC / Workload Identity Federation rather than a stored client secret, following the same no-stored-credential principle used for the application's own access to Storage and Key Vault. GitHub issues a short-lived signed token per workflow run; Azure AD trusts it for this specific repository (and, since the deploy job runs under a GitHub environment, for that environment specifically) and exchanges it for a short-lived Azure access token.

5. Known Limitations

Cost

The App Service Plan (B1) costs approximately €13/month. If terraform destroy is not run after the project is complete, the resources continue to run and incur charges. This is especially relevant for student subscriptions with a fixed budget limit.

No environment separation

The infrastructure currently only has a single environment (dev). In a production context, separate environments (e.g. staging, prod) would be used so that changes can be tested before affecting live data. The deployment pipeline likewise deploys directly to this single environment with no staging slot or blue-green deployment. For this project scope, a single environment is sufficient.

Remote Terraform state (addressed)

The Terraform state file (terraform.tfstate) was originally stored only locally on the developer's machine, meaning only the person who ran terraform apply could manage the infrastructure, and a lost file would mean Terraform losing track of all created resources. This has since been addressed with a bootstrap/ directory: a small, separate Terraform configuration that creates a dedicated Storage Account solely for holding the main project's state, solving the chicken-and-egg problem where the main Storage Account is created by Terraform but a remote backend needs a Storage Account to exist first. See bootstrap/main.tf and the commented-out backend "azurerm" {} block in providers.tf for the exact migration steps. The bootstrap configuration's own state remains local, since it consists of a handful of resources that essentially never change after creation.

Globally unique resource names (addressed)

Storage Account and Key Vault names must be unique across all of Azure, not just within this subscription. Relying on project_name alone risked a naming collision with someone else's resource of the same name. A random_string resource now generates a short suffix once (kept stable in the state file across subsequent applies) and appends it to both names, e.g. stcloudprojectdevab12c.

Consistent authorization model (addressed)

Storage access for the Web App's Managed Identity was already implemented as an Azure RBAC role assignment (Storage Blob Data Contributor). Key Vault access, however, originally used the older, Key-Vault-specific Access Policy model instead of RBAC — two different authorization systems within the same project. The Key Vault now has enable_rbac_authorization = true, and both the developer's own access (Key Vault Administrator) and the Web App's access (Key Vault Secrets User) are granted via azurerm_role_assignment, consistent with the Storage Account.

Deployment identity (addressed, optional)

By default, terraform apply is run authenticated as the developer's own Azure AD user via az login. As an alternative, spn-deployment-notes.md documents creating a dedicated Service Principal for Terraform deployments specifically — separate from the Service Principal used by the GitHub Actions pipeline, since the two serve different actors and purposes (a human running Terraform locally vs. an automated pipeline deploying application code). This is offered as an option rather than a requirement, since a single-developer student project does not strictly need this separation, but it demonstrates the pattern used in team/production settings.

Public blob container and unauthenticated upload

The images container is set to public blob access, meaning anyone who knows the URL of a file can access it directly, and the application's upload page has no access control — anyone with the application URL can upload or download files. For sensitive data or a multi-user production system, this would be a problem and would need application-level authentication (e.g. login, API keys, or a reverse proxy with access rules). For this project, which stores images intended to be displayed on a public web page, both decisions are intentional and within the agreed project scope.

Key Vault secret unused by the application

The Storage Account access key stored in Key Vault during Part I is not read by the Part II application, since the application authenticates to Storage via Managed Identity directly. It is retained to demonstrate that the secret-storage pattern from Part I still works correctly alongside the newer Managed Identity-based access added in Part II.

No monitoring

There are no alerts, diagnostics, or logging resources configured beyond the pipeline's own verify step and the application's /healthz endpoint. If the application crashes or unexpected usage occurs (e.g. a large number of uploads), there is no automated notification. In a production setup, Azure Monitor and Application Insights would be added to the infrastructure.
